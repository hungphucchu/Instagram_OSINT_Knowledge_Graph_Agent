# System Specification — Instagram OSINT Knowledge Graph Agent

> This document is the source of truth for the project. The TA feeds this file
> to an LLM during grading (`scripts/regenerate.sh`) and verifies that the
> regenerated source under `regenerated/src/myproject/` makes the user-story
> tests in `tests/user_stories/` pass. Anything outside this spec is
> implementation detail and may change without breaking the contract.

## 1. Purpose and Scope

The system is a six-phase Instagram OSINT pipeline that ingests public
Instagram-like post artifacts, extracts entities and relations, deduplicates
mentions into canonical entities, inserts the result into a Neo4j knowledge
graph, runs quality checks, and answers natural-language questions with
graph-grounded evidence.

The graded package is `src/myproject/`. It is a thin public facade over the
heavier internal implementation in `src/agents/`, `src/schemas/`, `config.py`,
and `logging_context.py`.

**Users**

- OSINT analysts exploring public Instagram activity
- Instructors and TAs grading the pipeline with fixtures and a live app
- Developers replaying the pipeline on deterministic sample data

**In scope**

- Offline fixture-driven ingest for CI and grading
- Offline dataset-driven ingest from `apify_data/` as the default starting point
- Optional live collection adapters behind a common abstraction
- Entity / relation extraction from caption-like text
- Deterministic deduplication with auditable clustering
- Idempotent graph insertion with provenance fields
- Quality validation over extracted and inserted data
- Graph-RAG natural-language query answering over Neo4j
- A minimal browser UI and HTTP API for the manual walkthrough
- Structured JSON logging with request-id propagation

**Out of scope**

- Private Instagram content, DMs, or bypass of access controls
- Real-time streaming ingestion
- Multimedia understanding of images, video, or audio
- Production auth / multi-tenant access control
- Query-time embedding retrieval; the graph is the retriever

### 2.3 Phase mapping

The phase documents under `phases/` describe the system incrementally. This
spec consolidates them into one contract:

- **Phase 0:** foundations, fixture dataset, configuration, provenance
- **Phase 1:** collection into a raw artifact store
- **Phase 2:** extraction into structured entity / relation records
- **Phase 3:** deduplication into canonical clusters
- **Phase 4:** graph insertion and linear LangGraph ingest
- **Phase 5:** quality checks and retry loop on the same LangGraph
- **Phase 6:** query answering over the graph

## 3. Data Flow

### 3.1 End-to-end flow

```text
fixture or apify_data input
  -> CollectionAgent
  -> RawArtifactStore
  -> ExtractionAgent
  -> ExtractionStore
  -> DedupAgent
  -> DedupStore / audit log
  -> GraphInsertionAgent
  -> Neo4j knowledge graph
  -> QualityAgent
  -> QueryAgent / Retriever / Generator
  -> HTTP API / browser UI
```

A diagram also lives at `docs/diagrams/architecture.svg`.

### 3.2 Phase 0 — Foundations

**Objective**

Provide a reproducible skeleton, deterministic fixture input, configuration
schema, run IDs, and provenance primitives so later phases can run without live
Instagram access.

**Key artifacts**

- `.env.example`
- `fixtures/raw_artifacts.json`
- `config.py` (`Settings`)
- `logging_context.py` / request IDs
- `schemas/raw_artifact.py`
- `schemas/provenance.py`

**Contract**

- The project must run from a flat `src/` layout.
- Shared config is environment-driven through `Settings`.
- Fixtures are valid input for sample ingest and tests.
- Every pipeline run can be identified by a `run_id`.

### 3.3 Phase 1 — Collection

**Objective**

Collect public-like artifacts from bundled offline inputs and persist them by
`run_id`.

**Inputs**

- Collection mode (`fixture` or `apify_data`)
- Run configuration: max items, seed handles, adapter settings

**Outputs**

- `RawArtifact` rows with at least:
  - `artifact_id`
  - `source_url`
  - `platform_post_id`
  - `caption_text`
  - `collected_at`
  - `run_id`
  - `collector_version`
  - `adapter_id`
  - `hashtags`
  - `mentions`
  - `raw_payload`

**Persistence**

- SQLite raw artifact store
- Optional local cache / replay files on disk

**Failure behavior**

- Fixture mode is offline and deterministic
- `apify_data` mode is offline and deterministic
- Paid live Apify API calls are not required for grading, reproduction, or
  normal development
- No secrets are logged

### 3.4 Phase 2 — Extraction

**Objective**

Read raw artifacts by `run_id` and emit structured extraction records.

**Inputs**

- Raw artifacts from Phase 1
- Extraction mode (`heuristic` or `llm`)

**Outputs**

- Extraction records containing:
  - `artifact_id`
  - `run_id`
  - `extractor_model_id`
  - entity list with type / surface form / snippet / confidence
  - relation list with subject / predicate / object / evidence span / confidence

**Persistence**

- SQLite extraction store

**Failure behavior**

- Malformed or empty artifacts do not crash the run
- Offline mode uses deterministic heuristics
- LLM mode is optional and configuration-driven

### 3.5 Phase 3 — Deduplication

**Objective**

Cluster duplicate or near-duplicate entity mentions into canonical entities with
an audit trail.

**Inputs**

- Extraction records for a single `run_id`

**Outputs**

- Dedup clusters and an audit log containing:
  - `canonical_id`
  - `canonical_surface`
  - aliases
  - mention IDs
  - threshold values used
  - merge rationale

**Persistence**

- SQLite dedup store
- Append-only audit records

**Failure behavior**

- Conservative false-split preference over unsafe false merges
- Deterministic char n-gram backend keeps CI offline

### 3.6 Phase 4 — Graph insertion

**Objective**

Insert canonical entities, posts, artifacts, and relationships into Neo4j with
idempotent upserts and provenance-friendly properties.

**Inputs**

- Raw artifacts
- Extraction records
- Dedup clusters

**Outputs**

- Neo4j nodes and relationships
- insertion summary counts

**Graph model**

- Nodes include `CanonicalEntity`, `Post`, and `Artifact`
- Relationships include graph facts such as:
  - `MENTIONS`
  - `TAGGED_IN`
  - `SOURCED_FROM`
- Stable identifiers prevent duplicate inserts on replay

**Provenance**

- `source_run_id`
- `artifact_id`
- `extractor_model_id`
- `snippet_hash`

**LangGraph orchestration**

- The ingest pipeline is one `StateGraph`
- Phase 4 is the linear slice:
  - `collect -> extract -> dedup -> graph_insert -> END`

### 3.7 Phase 5 — Quality checking

**Objective**

Judge extracted / inserted output and emit a `QualityReport`; optionally route
failed work back to `extract` or `dedup` on the same LangGraph.

**Inputs**

- Inserted graph and / or extraction-vs-source samples
- Quality configuration

**Outputs**

- `QualityReport` JSON
- optional quarantine JSON when critical issues exist

**Current baseline**

- Deterministic graph rule packs are not the primary path
- LLM-judge extraction mismatch findings are supported
- Quality node errors are wrapped as status-bearing results, not silent failures

**LangGraph extension**

- The same graph from Phase 4 gains:
  - `graph_insert -> quality`
  - conditional retry to `extract` or `dedup`
  - retry ceiling via config

### 3.8 Phase 6 — Query answering

**Objective**

Answer natural-language questions by treating the graph as the retriever.

**Inputs**

- User question
- live graph schema summary
- graph connection

**Outputs**

- grounded answer envelope:
  - `answer`
  - `evidence` / citations
  - `cypher`
  - `query_id`
  - `warnings`

**Current baseline**

- Query generation is deterministic-first with optional LLM translation
- Cypher is verified read-only before execution
- Empty evidence returns an explicit no-evidence answer
- The API returns rows summarized as citations

**Failure behavior**

- Mutating Cypher is rejected
- Result-set size is bounded by `QUERY_MAX_LIMIT`
- Missing LLM key produces a structured 503 only when the LLM is enabled

### 3.9 User-story flows

**US-01 happy path**

1. Browser or client POSTs `{"text": "...", "max_results": N}` to
   `POST /api/query`.
2. Request-id middleware allocates `req_xxxxxxxx`.
3. Router validates input and builds a query agent lazily.
4. Query agent derives or generates read-only Cypher, verifies it, executes it,
   and returns evidence rows.
5. Router converts rows into citations and returns the response envelope.

**US-02 empty input**

- `POST /api/query` with blank text returns HTTP 400
- no LLM call is required

**US-03 model key missing**

- if `QUERY_LLM_ENABLED=true` and `QUERY_LLM_API_KEY` is empty,
  the router raises `ModelNotConfiguredError`
- the API returns HTTP 503 with an operator-readable message

**US-04 sample pipeline**

- `POST /api/pipeline/sample` runs deterministic offline ingest starting from
  the bundled fixture and / or `apify_data/`
- it writes raw artifacts, extraction records, and dedup clusters
- it may attempt graph insertion if Neo4j is reachable
- graph insertion failure does not invalidate the count summary response

**US-05 graph stats**

- `GET /api/stats` reads node / edge counts from the live graph
- on graph read failure it returns zero counts rather than crashing

```text
GET /health
  -> 200 OK
  -> response body is the plain text string: "ok"

  -> 200 OK

     - a question input
     - a Submit button
```

### 4.3 Regeneration-critical notes

- `src/myproject/api.py` must expose a module-level FastAPI application object
  named `app`.
- `src/myproject/router.py` must expose a module-level helper `_agent()`
  because the user-story tests monkeypatch it.
- `route_query(...)` must call `_agent()` and expect the returned agent object
  to expose `answer(request)` where the response object has attributes
  `answer`, `evidence`, `cypher`, `query_id`, and `warnings`.
- `run_sample_ingest(...)` must be deterministic and work on the bundled
  fixture without requiring live LLM credentials.
- `GET /api/stats` must construct `Retriever()` inside the handler so tests can
  monkeypatch `myproject.retriever.Retriever`.

## 7. Model and Prompt Selection

### 7.1 Phase 2 extraction

The project supports two extraction paths:

1. **Heuristic extraction** for CI and offline runs
   - hashtags become topic-like entities
   - `@mentions` become person-like entities
   - simple relations such as `MENTIONS` and `TAGGED_IN` are emitted

2. **LLM-ready extraction** for live or higher-quality runs
   - OpenAI-compatible chat completion endpoint
   - structured JSON extraction into the same record schema

### 7.2 Phase 3 deduplication

Query-time embeddings are not used. Embedding-like signals belong only to
deduplication.

The baseline implementation uses deterministic char n-gram vectors so the
pipeline stays offline and reproducible. A stronger semantic backend is an
allowed extension, but it must preserve the same clustering contract.

### 7.3 Phase 5 quality

Quality checks may include an LLM judge that compares source text to extracted
records. Findings are written as structured violations in a `QualityReport`.

### 7.4 Phase 6 query answering

There are two model-assisted steps on the query path.

1. **NL -> Cypher translation**
   - prompt includes a live schema summary
   - output format is strict JSON with a `cypher` field
   - generated Cypher must be read-only and bounded

2. **Answer synthesis**
   - prompt includes the original question plus returned rows
   - output format is strict JSON with an `answer` field
   - if the model fails, contradicts the evidence, or is disabled, the system
     falls back to a deterministic summary

### 7.5 Safety and mitigation rules

- Generated Cypher is verified before execution
- Mutating clauses are rejected
- `LIMIT` is injected or clamped
- Empty evidence must produce an explicit no-evidence answer
- The system must never invent evidence that was not in the returned rows

### 7.6 Why Graph-RAG here

The target questions are graph-shaped, not chunk-retrieval shaped. Examples:

- who appeared together most often?
- which entities were mentioned in the same post?
- how many posts mention a topic or handle?

So the graph is the retriever, and the LLM is only a translator / summarizer
around graph execution.
