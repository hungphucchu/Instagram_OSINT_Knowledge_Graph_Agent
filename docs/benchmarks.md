# Benchmarks

> Performance numbers from the team's own load test run. The TA reproduces
> this on their hardware during Phase 1 grading.

## Hardware Profile

- CPU: Apple Silicon M2 (8 cores) / Intel x86_64 (8 cores)
- Memory: 16 GB
- Disk: SSD, 50 GB free
- Network: 100 Mbit (LLM endpoint over HTTPS)
- Container runtime: Docker 24.x
- Orchestration: Docker Compose v2

## Methodology

Load test driver: Locust 2.x (`tests/load/locustfile.py`).

Run command (also available as `make loadtest`):

```bash
locust -f tests/load/locustfile.py \
  --headless -u 20 -r 5 -t 60s \
  --host=http://localhost:8080 \
  --csv=reports/loadtest
```

- `-u 20` — 20 concurrent users
- `-r 5` — ramp up at 5 users/second
- `-t 60s` — sustained load for 60 seconds
- Headline endpoint: `POST /api/query`

Benchmark mode:

- For deterministic local benchmarking, run the app with `QUERY_LLM_ENABLED=false`
  so the upstream model call does not dominate latency.
- `make loadtest` regenerates both `reports/benchmarks.json` and
  `reports/loadtest_status.txt`.

## Headline Numbers

| Metric                              | Value          |
| ----------------------------------- | -------------- |
| Sustained throughput                | ~14 req/s      |
| Error rate                          | < 5%           |
| p50 latency (`POST /api/query`)     | ~180 ms¹       |
| p95 latency (`POST /api/query`)     | ~240 ms¹       |
| p99 latency (`POST /api/query`)     | ~410 ms¹       |
| `GET /health` p95                   | < 10 ms        |
| Peak memory (app container)         | ~720 MB        |
| Cold start time                     | ~6 s           |

¹ Measured with `QUERY_LLM_ENABLED=false` so LLM latency does not dominate.
With the live Qwen3-8B endpoint, p95 is dominated by the upstream call
(~1.5 s typical).

Rubric thresholds (Stress and Robustness category):

- ≥10 req/s sustained
- < 5 % error rate over a 60-second window

Raw timings: `reports/benchmarks.json` (regenerated by `make loadtest`).
Pass/fail summary for the grader: `reports/loadtest_status.txt`.
