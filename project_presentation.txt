Project Presentation (5%)
Title: Instagram OSINT Knowledge Graph Agent (Pure GraphRAG)

Slide 1 - Problem
- OSINT-style social data is noisy and disconnected.
  - Noisy: informal language (slang/emojis), typos, inconsistent naming (@alex vs Alex vs “my friend A”), weak or ambiguous signals, and incomplete metadata.
  - Disconnected: evidence is scattered across many posts; relationships are implied in text (not stored as explicit links); the same real-world entity repeats without a shared stable ID.
- It is hard to answer relationship-heavy questions using flat text search.
  - Keyword or “similar text” retrieval returns a few chunks, but it does not reliably preserve who is connected to whom across posts.
  - Multi-hop questions (for example, co-appearance patterns) require stitching many separate mentions manually.
- Need a traceable pipeline that connects entities and evidence.
  - Every structured fact should point back to the source artifact/run and the extraction/dedup decisions that produced it.
  - This supports explainable answers and reduces “guessing” from unstructured text alone.

Slide 2 - Goal
- Turn noisy Instagram-like text into a trustworthy, queryable knowledge graph.
- Deliver user-friendly answers that are grounded in graph evidence (not hallucinated text).
- Ensure traceability and trust: every answer can be linked back to source artifacts and pipeline decisions.
- Provide an interactive product experience (chat, agent runner, graph view), not only offline scripts.

Slide 3 - Why GraphRAG
- Query-time retrieval is graph traversal (not embedding-only retrieval).
- Better for multi-hop questions (who appears with whom, what post mentions whom).
- More explainable because answers map to explicit nodes/edges.

Slide 4 - Architecture (brief overall)
- Main ingest flow (build the graph, orchestrated with LangGraph):
  Collection -> Extraction -> Deduplication -> Graph Insertion
- Quality flow (separate validation path after graph insertion):
  Quality checks read graph + pipeline outputs and produce report/quarantine artifacts.
- Query flow (separate user-facing QA path):
  User question -> QueryAgent -> safe Cypher -> Neo4j -> answer + evidence.
- Storage split by purpose:
  SQLite stores staged pipeline artifacts; Neo4j stores the queryable graph.
- UI/Backend integration:
  Next.js frontend calls API routes, and API routes call Python CLI services.

Slide 5 - Agent 1: Collection
- Input: public Instagram-like post content from Apify API (live mode) or local fixtures (offline mode).
- Output: raw artifacts in SQLite (evidence layer).
- Role: preserve original text/metadata for provenance and replay.
- Example: caption "Congrats @alice and @bob at NASA event" is stored as one raw artifact with source metadata.

Slide 6 - Agent 2: Extraction
- Input: raw post text.
- Output: structured entities + relations.
- Method: heuristic/LLM extraction with normalized schema.
- Example: extract entities {Alice, Bob, NASA} and relation {Alice MENTIONS Bob}.

Slide 7 - Agent 3: Deduplication
- Input: extracted mentions/entities.
- Output: canonical entity IDs + merge audit decisions.
- Method: fuzzy matching + deterministic merge/review thresholds.
- Example: "@nasa", "NASA", and "N.A.S.A" merge into one canonical entity ID.

Slide 8 - Agent 4: Graph Insertion
- Input: canonical entities + relations + provenance.
- Output: Neo4j nodes/edges with idempotent upserts.
- Role: build the queryable knowledge graph.
- Example: create nodes (Alice, Bob, NASA) and edges MENTIONS / CO_APPEARS_WITH with run_id provenance.

Slide 9 - Agent 5: Quality
- Input: graph slice + pipeline outputs.
- Output: quality report + optional quarantine file.
- Method: deterministic integrity checks first, then LLM semantic judge.
- Example: flag orphan nodes or conflicting duplicate links, then write `quality_*.json` report.

Slide 10 - Agent 6: Query (GraphRAG)
- Input: user natural-language question.
- Flow: NL -> LLM Cypher -> Query Safety Guard (read-only checker) -> Neo4j execute -> LLM answer synthesis.
- Output: friendly answer + evidence rows + cypher + warnings.
- Reliability hardening: malformed Cypher repair, JSON parsing robustness, contradiction guard.
- Example: question "Who appeared together most often?" returns top pair + evidence rows + executed Cypher.

Slide 11 - Frontend + Backend App (completed)
- Frontend (Next.js/React/TypeScript) has 3 pages:
  1) Chat Interface
  2) Run Agents (whole pipeline or per stage)
  3) Graph Display (relationship overview)
- Backend bridge:
  - `/api/query` for QueryAgent
  - `/api/agent-run` for stage/pipeline commands

Slide 12 - Data and Provenance
- Core entities: Person, Organization, Location, Post, Artifact
- Core edges: TAGGED_IN, MENTIONS, CO_APPEARS_WITH, SOURCED_FROM
- Provenance fields include source_run_id, snippet_hash, extractor_model

Slide 14 - Demo Flow
1) collect fixtures
2) extract entities/relations
3) dedup into canonical ids
4) graph_insert into Neo4j
5) run quality gate and inspect report/quarantine outputs
6) ask NL question in chat UI -> get answer + cypher + evidence
7) visualize relationship evidence in graph page

Slide 15 - Validation
- Lint and tests pass
- Rerun idempotency confirmed (stable graph counts)
- Provenance retrievable on graph evidence paths
- Query safety checks block mutating/invalid Cypher
- Query reliability hardening done:
  - robust parsing for noisy LLM JSON
  - contradiction guard when answer says “no evidence” but rows exist
  - auto-repair for common malformed Cypher patterns (e.g., `COUNT({ ... })`)

Slide 16 - Testing Strategy (dedicated)
- Unit tests:
  per-agent logic tests (extraction, dedup, quality rules, query safety guard) using deterministic fixtures/mocks.
- Integration tests:
  pipeline path tests (collect -> extract -> dedup -> graph insert -> quality) with fixture data.
- Query regression tests:
  verify Cypher guard blocks mutations, malformed query handling, and answer/evidence consistency checks.
- Frontend/backend smoke tests:
  UI calls Next.js API routes, which trigger Python CLI and return parseable outputs (answer, cypher, evidence).
- Quality gates in development:
  `pytest` + `ruff` run before merges/demos to reduce runtime surprises.

Slide 17 - Next Steps
- API hardening for production use (auth, rate limits, request validation)
- Better graph visualization UX (interactive zoom/filter/grouping)
- Stronger query semantics for vague asks (e.g., “most famous” definition policy)
- Optional dual-model Cypher drafting + verification

Slide 18 - Limits, Scope, and Cost
- Upfront cost (before launch):
  initial setup effort for cloud infrastructure, CI/CD, domain/SSL, and environment hardening.
- Data collection cost (Apify, ongoing):
  pulling larger volumes or frequent refresh jobs increases usage-based fees; this prototype keeps runs bounded for cost control.
- LLM inference cost (ongoing):
  extraction/query generation/answer synthesis add per-call token cost and latency at scale.
- Application hosting cost (ongoing, e.g., GCP):
  frontend hosting + backend runtime + logging/monitoring contribute monthly spend.
- Online graph database hosting cost (ongoing):
  managed Neo4j/Aura or self-hosted Neo4j VM/storage/backup/network egress all add recurring cost.
- Current project scope:
  classroom/research prototype with controlled data volume, not yet a fully optimized high-throughput production deployment.

Slide 19 - Expected Outcome
- End-to-end, explainable GraphRAG pipeline
- Natural-language answers grounded in structured graph evidence
- Reusable foundation for course demos, interactive web demos, and future research extensions
