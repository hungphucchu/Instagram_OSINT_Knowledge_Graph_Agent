Instagram OSINT Knowledge Graph Agent
Multi Agent Pipeline with Entity Extraction and Graph Persistence
Students build a pipeline of six cooperating agents that collects publicly available data from Instagram, extracts
named entities and relationships from the content, resolves duplicates using fuzzy string matching and semantic
embeddings, and stores everything in a knowledge graph with full provenance tracking.
The final graph supports natural language queries, allowing a user to ask questions like "who appeared together
most often" and receive answers grounded in the extracted data. Each agent handles one stage: collection,
extraction, deduplication, graph insertion, quality checking, and query answering.


This is an incredibly robust and highly relevant architecture. You are effectively designing an advanced **Graph RAG (Retrieval-Augmented Generation)** system paired with a multi-agent orchestration layer. 

### **The Verdict on the Project**
Moving beyond standard vector-based RAG to a structured Knowledge Graph is a massive leap in AI architecture. Standard RAG struggles with global reasoning (e.g., "who appeared together most often"), whereas this graph-based approach excels at it. 

**The biggest bottleneck will not be the AI, but the data collection.** Instagram has notoriously aggressive anti-scraping mechanisms. The agents will need to handle rate limits, captchas, and dynamic DOM changes gracefully. However, once the graph is populated, it perfectly stages the data for advanced downstream tasks like link prediction, anomaly detection, or applying Graph Neural Networks to map out hidden social clusters.

Here is a breakdown of the best modern tooling to implement this six-agent pipeline.

---

### **1. Agent Orchestration Layer**
Before defining the individual agents, you need a framework to manage their state, handle handoffs, and manage the cyclical nature of quality checking.
* **LangGraph:** The absolute best choice for this. It allows you to model the pipeline as a state machine (a directed cyclic graph). If the "Quality Checking" agent detects an error, LangGraph easily routes the flow back to the "Extraction" agent.
* **CrewAI:** A slightly higher-level alternative that assigns roles and goals to agents, though it offers slightly less granular control over the exact state flow than LangGraph.

### **2. The Six-Agent Implementation Stack**

#### **Agent 1: Collection (The OSINT Scraper)**
* **Apify (Instagram Scraper API):** Highly recommended over writing a custom scraper. Apify handles the proxy rotation and session management required to reliably extract Instagram posts, comments, and tags.
* **Instaloader (Python):** If you want an open-source, local approach. Be prepared to carefully manage session cookies to avoid account bans.

#### **Agent 2: Extraction (NER & Relationships)**
* **Instructor (Python library) + LLM:** Instructor patches an LLM (like `gpt-4o-mini` or Claude 3.5 Sonnet) to force it to output strict, validated JSON (e.g., extracting entities as `Person`, `Location`, `Organization`, and relations as `APPEARED_WITH`).
* **GLiNER (Generalist and Lightweight Model for Named Entity Recognition):** A smaller, highly efficient open-source model specifically fine-tuned for extracting entities if you want to save on LLM API costs.

#### **Agent 3: Deduplication & Resolution**
* **SentenceTransformers (`all-MiniLM-L6-v2`):** For generating semantic embeddings of the extracted entities (e.g., recognizing that "The Met" and "Metropolitan Museum of Art" occupy the same semantic space).
* **RapidFuzz:** A fast string-matching library for the fuzzy matching component (catching typos or slight handle variations like `@john_doe` and `john doe`).

#### **Agent 4: Graph Insertion & Persistence**
* **Neo4j:** The industry standard for graph databases. It handles provenance tracking beautifully via relationship properties (e.g., an `EXTRACTED_FROM` edge linking an entity back to the original Instagram post URL). 
* **AuraDB:** Neo4j's fully managed cloud service, which integrates seamlessly with Python via the `neo4j` driver.



#### **Agent 5: Quality Checking (The Judge)**
* **LLM-as-a-Judge:** Use a highly capable model to sample the extracted JSON and the original text to verify accuracy. 
* **Guardrails AI:** An open-source framework that can sit on top of your extraction agent to enforce data validation (e.g., ensuring no entity is inserted without a valid source URL).

#### **Agent 6: Query Answering**
* **LangChain's `GraphCypherQAChain`:** This chain takes a user's natural language question, translates it into a Cypher query (Neo4j's query language), runs it against the database, and uses the resulting graph data to generate a conversational response.

---

### **Implementation Strategy**
Scaffolding the agent handoffs and API integrations is a rapid process when leveraging autonomous coding environments to generate the boilerplate, allowing you to focus purely on the system design and prompts. I recommend building this in reverse: set up the Neo4j schema first, manually insert dummy data, and build the Query Answering agent. Once you prove the graph can answer complex questions, build the pipeline that feeds it.

Are you planning to run the language models locally for privacy/cost control, or will you be relying on commercial APIs like OpenAI or Anthropic for the extraction and querying agents?
