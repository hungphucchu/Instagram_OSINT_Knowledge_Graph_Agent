Philosophy: From Vibe Coding to Disciplined Engineering
Agentic LLMs have changed what it means to write software. A capable model can generate working code from a paragraph of intent. This is
powerful, and it is also dangerous. The dangerous version is what the community calls vibe coding: a developer types a vague description
into an LLM, accepts whatever comes back, runs it once, and moves on. The code may work. The developer may not know why. There is no
specification, no tests tied to requirements, no plan that survives the first surprise.
Vibe coding does not scale. It does not survive code review, on-call rotations, security audits, customer escalations, or a teammate trying to
extend the codebase six months later. It also does not survive automated regeneration: ask an LLM to rebuild the system from a vague
description and you get a different system every time. The research community now treats specification driven development as the discipline
that keeps agentic LLMs trustworthy in production.
This project asks you to practice that discipline end to end. The rubric is organized around the six pillars of the software development
lifecycle. Each pillar pairs the engineering practice with the artifact you produce, and the test that verifies you produced something real and
not theatrical.
The six pillars
Plan. Decide what you are building and for whom, before you write code. In this rubric: a Specification document with public interfaces, and
User Stories with acceptance criteria and manual walkthrough steps. The TA verifies your plan is real by feeding it to an LLM and checking
that the regenerated code passes your story tests.
Design. Decide how the system will be reproducible by anyone but you. In this rubric: pinned versions, documented data and model
provenance, a single command full replay. The TA runs that one command on a clean machine and checks your reported numbers come
back.
Implement. Write code that other engineers can read and that does not introduce known vulnerabilities. In this rubric: linting, formatting,
type checking, dependency scanning, and a model card that names the system's limits. The TA runs the same checks and reads the model
card.
Test. Verify behavior against requirements automatically. In this rubric: unit, integration, user story acceptance, edge, and load tests, all
green, with coverage at threshold. Every user story has both an automated test and a manual walkthrough; the two together are how
requirements become contracts.
Deploy. Make the system runnable by someone who does not have your laptop. In this rubric: a Dockerfile, a Compose file, a documented
environment contract, and a quick start in the README that the TA executes literally. Zero undocumented manual steps.
Operate. Hand a working system to a real user and let them use it. In this rubric: structured logs the TA can trace, a UI a TA can walk
through using only your STORIES document, and documentation that a cold reader can follow without asking the team. This is where most
points live, because this is where users live.

What this project teaches you
Working with agentic LLMs as a force multiplier requires the same disciplines that make any software system trustworthy: a written contract
before code, a reproducible build, a real test suite, a clear deployment story, observable runtime, and accessible documentation. The rubric
makes those disciplines countable. Each TEST row is an executable check that distinguishes work from theater.
If you find yourself tempted to write code first and document later, notice what happens at grading time: the spec regeneration test feeds
your spec to the same kind of LLM you used while building. If your spec is decoration, regeneration produces a system that fails your own
user story tests, and the largest single category in the rubric scores zero. The shortest path to a good grade is also the shortest path to a
system you would actually deploy.
How Grading Works
The TA clones your public GitHub repository on a clean machine and works through three phases. About two thirds of the points are
computed by automated scripts; the rest comes from a manual walkthrough against your running app. You can run all of these checks
yourself with `scripts/preflight.sh` before pushing, and your local result is exactly what the TA will see.
Phase 1. Automated checks (69 pts). make reproduce, make test, make lint, pip-audit, make loadtest, scripts/regenerate.sh. Each writes a
JUnit XML or text report under reports/. The grading script reads those reports and scores Plan, Design, Test (both categories), Implement,
and Team.
Phase 2. Bring up the system (5 pts for Logging trace). docker compose up reaches healthy under 10 minutes. The TA traces one live
request through docker compose logs by request ID; this scores the Logging category.
Phase 3. Manual walkthrough (26 pts). The TA opens docs/STORIES.md and walks every story step by step against the live UI, recording
pass or fail per story. The TA also confirms the README cold-read works and that every story has a usage section. This scores App
Functionality (20) and Documentation (6).
Key Terms
Spec regeneration. Feeding docs/SPEC.md to an LLM with the course-issued prompt, then running your user story tests against the
generated code. If the spec is the real contract, the generated code passes; if the spec is decoration, it does not.
User story. A short feature description from the user's perspective with a stable ID (US-NN), Given/When/Then acceptance criteria, and
numbered manual steps a TA can follow without reading source. Stored in docs/STORIES.md.
Manual walkthrough. The TA opens STORIES.md and follows each story step by step against the running app, recording pass or fail in
reports/walkthrough.md. This is how App Functionality is graded.
Traceability matrix. grading/traceability.yaml. Links each user story to its spec section, source modules, and tests, so the grader can detect
orphans.
Preflight. scripts/preflight.sh. Runs every automated check the TA will run. If preflight passes locally, your automated grade passes. The
manual walkthrough still requires a human.

How to Submit
• Push the entire project to a public GitHub repository owned by your team. Submit the repository URL through the course submission
system on or before May 10, 11:59 PM. The link must point to a public repo so the TA can clone without credentials. Repos that are
private, deleted, or made private after submission score zero.
• The grading script reads commit_sha from grading/manifest.yaml. If missing or stale, the TA grades the latest commit on main as of the
deadline.
• Run scripts/preflight.sh locally before pushing. If preflight fails, your grade fails the same way.
• Late submissions follow the course-wide late policy. Contact the instructor before the deadline if you need an extension; post-hoc
explanations are not accepted.
